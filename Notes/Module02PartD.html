Grammars, Intepreters, and Abstract Syntax Trees

Grammars and Intepretation

A few very general points: * Language structure is defined by the rules of a grammar - There is serious computer science theory revolving around the concept of a grammar - With some very specific limitations put on programming languages to tie us to a specific type of grammar * Program syntax is defined by a grammar * Parsing: the process of determining whether a program’s syntax is correct by applying the rules of a grammar - Top-down: Start with the most general production and work our way down to generate the code being checked - Bottom-up: Start with the statements being checked and see if we generate productions that will get us to the top level

The Interpreter Design Pattern

Interpreter Pattern: A design pattern that specifies how to evaluate sentences in a language. * The idea is to provide a class for each symbol in a grammar. * A sentence is parsed by creating a syntax tree that is an instance of the composite pattern that can be used to interpret that sentence. * Composite pattern: A design pattern that describes a group of objects s.t. one may treat the composite as instance of the same type as the objects in the group

Abstract Syntax Trees

Abstract Syntax Tree (AST): A tree representation of the abstract syntactic structure of text written in a formal language such as a programming language

Interpreter Class and Object Diagram

The Client class uses the AbstractExpression interface’s interpret(context) method. The TerminalExpression class is final and interprets an expression directly. This class is the representation of the terminals in the grammar. The NonTerminalExpression is the composite instance in the design and maintains a container of child expressions expxpressons and forwards interpret request these expressions.

The Client object maintains a context in which to evaluate expressions. The parsing process builds an AST that is a composite object of type AbstractExpression. The intrerpret method tries to traverse the AST based on the contains of the context.

A Two Step Process

The Parsing Process

Let’s look a case study of building a parser for prefix integer expressions.

The Grammar

<expression> ::= <number>
             ::= (identifer>
             ::= <primitive> ( {<expression>}*(,) )
<primitive>  ::= + | - | * | add1 | sub1

For example:

3 
+(3,2)
-(3, add1(2))
x
+(3,x)

For the moment we will assume the semantics one intuitively expects for integer arithmetic.

Abstract Syntax

<expression> ::= <number>
             (lit-exp (datum))
             ::= (identifer>
             (var-exp (id)
             ::= <primitive> ( {<expression>}*(,) )
             (primapp-exp (prim rands)

Now we fall back to our textbook extensions to Racket:

(define-datatype expression expression?
   (lit-exp (datum number?))
   (var-exp (id symbol?))
   (primapp-exp 
     (prim primitive?)
     (rands (list-of expression?))))

And for the primitive production in the gramamar:

(define-datatype primitive primitive?
   (add-prim)
   (subtract-prim)
   (mult-prim)
   (incr-prim)
   (decr-prim))

where each of the primitive functions are build using the internal Racket implementation of each primitive.

Evaluating Expressions: Interpreter Interface

(define exp (primapp-exp (add-prim (list (lit-exp 2) (var-exp 'x)))
(eval-expression exp env)

So, what’s is env. That’s the context from the Interpreter pattern. We need to consider how our interpreter will represent that data

The Environment Datatype

Consider again the Interpreter pattern: * Expression evaluation (the interpret method from the AbstractExpression interface) occurs in a context maintained by the Client object * The grammar used to build an AST has the concept of identifier that needs to be mapped into a value * Those values are extracted from the context

What Does Prior Experience Tell Us?

Consider the following C++ program:

#include <iostream>
int square(int aValue) {
    return aValue * aValue;
}

int main() {
   int accumulator = 0;
   for (int i = 0; i <=9 ; ++i) {
      accumlator += square(i);
      std::cout << accumulator << std::endl;
    }
}

We’ve talked in the past about variables being in “scope”: global or local. We’ve also discussed the concept of stack frames, which is how we address function calls. This gives us the intuition regarding context in an interpreter.

Getting Mathy With It

Here we see the first example of creating a datatype to capture behavior * From our C++ example, starting with the global scope, we have an “environment” that defines context at a point in time * With the system stack being used as a container for that enviornment.

Environment: A function whose domain is a finite set of variables that has a range from the set of all values understood by an implementation language * For our purposes, we will define the range of the environment to be set of all Racket values * In C++, this is implicit within the language runtime * We denote the value of a variable var in an environment env as its binding in env

The interface to the environment data type is specified as follows:

(empty-env)            = [0]
(apply-env [v] var)    = f(var)
(extend-env var v [f]) = [g]

where g(var1) = v, if var1 = var or g(var1) = f(var1) otherwise.

An example will claarify:

(define e
    (extend-env 'd 6
      (extend-env 'y 8
         (extend-evn 'x 7
             (extend-env 'y 14
                (empty-env))))))

where in the environment e is such that e(d) = 6, e(x) = 7, e(y) = 8 and e is undefined on any other variables. Note how the binding of y to 14 is overridden by a later bind to 8.

Note how this works: Every environment can be built by starting with the empty environment and applying extend-env for some number of times. Thus, we have the grammar:

Env-exp ::= (empty-env)
        ::= (extend-env Identifier Racket-value Env-exp)

Note that we more than just a functional representation: the grammar can just as well describe a set of lists:

;;; Env = (empty-envn) | (extend-env Var SchemeVal Env) 
;;; Var = Sym

;;; empty-env : () → Env 
(define empty-env
  (lambda () (list ’empty-env)))
  
;;; extend-env : Var × SchemeVal × Env → Env 
(define extend-env
    (lambda (var val env)
    (list ’extend-env var val env)))

;;; apply-env : Env × Var → SchemeVal 
(define apply-env
  (lambda (env search-var)
    (cond
        ((eqv? (car env) ’empty-env) 
            (report-no-binding-found search-var))
        ((eqv? (car env) ’extend-env) 
            (let ((saved-var (cadr env))
                  (saved-val (caddr env))
                  (saved-env (cadddr env))) 
              (if (eqv? search-var saved-var)
                  saved-val
                  (apply-env saved-env search-var))))
              (else
                  (report-invalid-env env)))))

(define report-no-binding-found
  (lambda (search-var)
    (eopl:error ’apply-env "No binding for ~s" search-var)))
    
(define report-invalid-env
  (lambda (env)
    (eopl:error ’apply-env "Bad environment: ~s" env)))

Or we can build a functional representation:

;;; empty-env : () → Env 
(define empty-env
    (lambda ()
      (lambda (search-var)
          (report-no-binding-found search-var))))
          
;;; extend-env : Var × SchemeVal × Env → Env 
(define extend-env
    (lambda (saved-var saved-val saved-env) 
        (lambda (search-var)
            (if (eqv? search-var saved-var) 
                saved-val
                (apply-env saved-env search-var)))))
                
;;; apply-env : Env × Var → SchemeVal 
(define apply-env
    (lambda (env search-var)
      (env search-var)))

Note how we’ve changed empty-env and extend-env to return callable forms that, when applied, do the same that that append-env did in the set of lists implementation. extend-env now returns a new procedure that represents the extended-environment.

Another Design Pattern

The case of a data type with a single observer is a common pattern. The pattern can be implemented by: * Identifying the lambda expressions in the client code whose evaluation yields values of this data type * Create a constructor form for each such lambda expression * Parameters will be the free variables in the lambda expression * Replace each such lambda expression in the client code with an invocation of the corresponding constructor. * Define an apply- procedure like apply-env in our environment interface. * Find all places in the client code, including bodies of the constructor procedures, where a value of the type is applied * Replace each such application by an invocation of the apply- procedure
